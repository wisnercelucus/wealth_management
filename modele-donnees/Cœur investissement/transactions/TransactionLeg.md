# TransactionLeg

The **financial truth** — one row per movement. A leg says what moved, where, signed,
and what it did to the cost memory. Positions, lots, and every P&L stream derive from
legs and legs only; nothing here is ever recomputed from the header.

## Essential fields

| Field | Type | Req | Description |
|-------|------|-----|-------------|
| `id` | uuid | ● | Surrogate primary key. |
| `transactionId` | FK→Transaction | ● | The event this movement belongs to. |
| `portfolioId` | FK→Portfolio | ● | **The address** — a client book, a fund/NAOS book, or a house book. Resolves the owner, the bank account, the GL mapping. |
| `instrumentId` | FK→Instrument | ● | What moves. **Cash is an instrument** — a cash leg and a security leg have the same shape; the difference is data, not structure. |
| `amount` | decimal(28,8) | ● | The one number — units, nominal, or money in the instrument's unit. **The sign is the direction.** `CHECK (amount != 0)`: a conditional leg that doesn't apply is *omitted*, never written as zero. |
| `currency` | char(3) | ● | Denomination **snapshot** — instrument currency is mutable reference data; a ten-year-old leg must not change meaning if it is ever corrected. Deliberately a `char(3)` code, not a `currencyId` FK: the snapshot must not follow reference-data edits (the one sanctioned break of the join-on-id convention; Lot and TransactionAmount share it). |
| `lotEffect` | enum | ● | What this leg does to lots — see below. |
| `principalFlag` | enum | ○ | `PRINCIPAL \| OTHER \| NULL` — see below. `CHECK (principalFlag IS NULL OR lotEffect = 'CARRY')`. |
| `amountName` | enum | ○ | Provenance — which [TransactionAmount](TransactionAmount.md) row fed this leg (nine-name enum). Documentation-by-reference; **never read downstream**. |
| `chargeNameId` | FK→ChargeName | ○ | **Charge marker** — set iff this leg is a charge leg, generated by the engine (a declared charge field or a policy), never by a contract line. One column, two jobs: `chargeNameId = X` filters a charge for analysis; `IS NOT NULL` is the "is a charge leg" boolean. `CHECK (amountName IS NULL OR chargeNameId IS NULL)` — witness-fed or charge, never both. See the [Charges module](../charges/README.md). |
| + envelope | | ● | See [README](README.md). Append-only: legs exist iff their transaction is `POSTED`, written once; `isActive` CHECK `true`. |

## lotEffect — what the leg does to lots

Lots are the only stateful store; a leg can create one, consume some, leave them alone,
adjust basis, or restate them. Closed by construction — there is no sixth thing to do to
the store:

- **`ESTABLISH`** — creates a [Lot](Lot.md): quantity in, basis remembered. Buy,
  subscription, dividend-in-shares (at stated basis, no cash leg required).
- **`RELIEVE_REALIZE`** — consumes lots **and** freezes realized P&L (the double name is
  honest: two outputs, one operation). Sell, redemption, maturity, amortization.
  Triggers the disposal walk, which writes [LotRelief](LotRelief.md).
- **`CARRY`** — **no lot event**: nothing created, nothing consumed, basis carried
  intact. Cash (no basis exists), income, provisions, charge settlements. `CARRY` never
  meant "cash"; it means "no lot event."
- **`ADJUST_BASIS`** — changes **basis without quantity**: capital return. Defined,
  **dormant in MVP**; when it wakes, it writes append-only basis-adjustment rows and
  decrements `Lot.costRemaining` — never rewrites `unitCost`.
- **`RESTATE`** — changes **quantity or location, never cost, never the holding clock**:
  split (quantity-delta legs), securities transfer (− at source, + at destination). Runs
  the restate routine: parent quantity closes into child lots carrying the parent's cost
  share, `openDate` and `fxAtOpen` (`parentLotId` lineage). No cash, no P&L. See
  [Lot](Lot.md).

## principalFlag — protecting the basis

Exists for exactly one situation: **two cash legs, same portfolio, same cash instrument,
same `CARRY`, in one transaction — and the lot engine must treat them differently.** A
bond buy pays −490,000 (the lot's price) and −8,082 (reimbursing the seller's earned
interest, offset at the next coupon). Merged, the lot costs 99.62 instead of 98 and
every future P&L is wrong.

- **`PRINCIPAL`** — the lot event's own cash: the basis at `ESTABLISH`, the proceeds at
  `RELIEVE_REALIZE`.
- **`OTHER`** — rides beside the lot event, settles later (transacted accrued).
- **`NULL`** — nothing for the question to apply to: deposits, income, FX, charge
  settlements, and every non-`CARRY` leg (a `RESTATE` moves no cash of its own).

Interest-income netting falls out free: Σ(`INCOME` legs) + Σ(signed `OTHER` legs) = the
period's true interest income — two sums, zero logic.

## Deliberately absent

- **Price** — derivable: `PRINCIPAL` cash ÷ `ESTABLISH` quantity (the lot builder's own
  formula).
- **Debit/credit** — the journal template's language, at the Zoho boundary only. Sign is
  direction here.
- **Date / status** — the transaction's; legs exist iff it is `POSTED`.
- **Counterparty, issuer, bank account** — resolved through the portfolio and instrument
  FKs. *The transaction is the event; the legs are its addresses.*
- **Memo** — narrative lives on the transaction; a leg needing its own story is a smell
  that it should be its own transaction.

## Notes & rules

- **Converted settlements**: when the settlement cash instrument's currency differs from
  the deal instrument's, each cash leg carries its source amount × the header `fxRate`
  (`PRINCIPAL × fxRate`, `ACCRUED × fxRate`, `INCOME × fxRate`) and Σ of those legs =
  the `COUNTER` witness. Fully reconstructible, so no per-leg converted witnesses exist.
- **Effective date is a rule, not a column**: CASH-kind legs take effect on
  `settleDate`, all others on `tradeDate` — derived from the leg's instrument kind plus
  the header dates. A stored `effectiveDate` would be a banned derivable.
- **Charge legs are auto-generated** at posting — never authored as contract lines.
  One leg per charge value: − on the settlement book of the header's `portfolioId`
  (the `TRANSACTION_CASH` resolution; the book itself when it self-settles), in that
  book's declared cash instrument (`Portfolio.cashInstrumentId`), stamped
  `chargeNameId`. Always `CARRY` with `principalFlag NULL`, so the lot builder's
  `CARRY + PRINCIPAL` selector never matches them — **charges never enter cost
  basis**. No house book receives a leg: the receiving side (revenue, tax payable) is
  the Accounting module's product at the GL, bound by charge name.
- Indexes: `(portfolioId, instrumentId)` — positions, reconciliation ·
  `(transactionId)` — event assembly · partial `(instrumentId) WHERE lotEffect NOT IN
  ('CARRY')` — the walk's lot-event scan · partial `(chargeNameId) WHERE chargeNameId
  IS NOT NULL` — charge analysis.
- Reversal legs are new rows under the reversing transaction — never edits.

## Clean model

```
TransactionLeg
  id             uuid    PK
  transactionId  FK Transaction
  portfolioId    FK Portfolio
  instrumentId   FK Instrument
  amount         decimal(28,8)   -- signed; sign is direction; != 0
  currency       char(3)         -- snapshot
  lotEffect      enum (ESTABLISH | RELIEVE_REALIZE | CARRY | ADJUST_BASIS | RESTATE)
  principalFlag  enum (PRINCIPAL | OTHER)?   -- NULL unless lotEffect = CARRY
  amountName     enum?            -- nine-name enum; provenance only
  chargeNameId   FK ChargeName?   -- set iff charge leg (auto-generated); never with amountName
  + envelope (isActive, externalId, externalRef, createdAt, createdBy, modifiedAt, modifiedBy)
  -- append-only: isActive CHECK true; modifiedAt/By stay NULL
```
